# 灰豚数据入库方案

> 本文档记录从千瓜插件入库迁移到灰豚离线下载入库的讨论结论，供后续开发导入脚本时参考。  
> 数据样本时间窗口：**2026-04-29 ~ 2026-06-20**

---

## 1. 背景与目标

### 1.1 原有方案（千瓜 + Chrome 插件）

在千瓜网站通过油猴脚本拦截 API，按品牌逐页采集商业笔记，再补全笔记详情，写入 Supabase 同一套数据库（`qiangua_brand` / `qiangua_blogger` / `qiangua_note_info`）。

### 1.2 新方案（灰豚离线下载）

灰豚平台支持批量导出，速度远快于网页翻页。导出两类文件后，**按笔记 ID 关联**，即可拼出一条完整的「品牌 + 投放 + 笔记详情」记录，仍写入上述三张表，下游报告、AI 分析、商品关联等逻辑无需改表结构。

含 3 个 Sheet：

#### Sheet `笔记`（458 行）— 核心详情

偏「内容 + 实际互动」视角：

- **互动（实际）**：收藏、评论、分享、点赞、互动、阅读数

内容完整度：458 条均有 50 字以上正文、图片列表；87 条有视频链接。

#### Sheet `博主`（432 行）

达人档案：用户ID、小红书ID、昵称、头像、个人描述（常含邮箱）、IP属地、官方认证、互动、粉丝、关注、采集更新。

#### Sheet `评论`（0 行）

当前为空，未导出评论数据。

### 2.4 两类文件的关系

```
5 个品牌 CSV  ──(同一批 458 条笔记 ID)──►  小红书笔记详情.xls

CSV 侧重：品牌 / 报价 / 达人等级 / 预估阅读
XLS 侧重：正文 / 图片 / 标签 / 实际阅读 / 博主档案
```

入库时以 **笔记 ID** 关联：CSV 补投放与互动，XLS 补正文与媒体（互动数不用 XLS）。

### 2.5 技术注意点

1. `小红书笔记详情.xls` 用 Python `pandas` / `xlrd` 可能报 UTF-16 编码错误；项目内 **Node `xlsx` 包（0.18.5）可正常读取**。
2. CSV 封面图为小红书 CDN（`sns-img-hw.xhscdn.com`）；XLS 图片多为灰豚 OSS（`cod-resource.oss-cn-shanghai.aliyuncs.com`），入库时需统一 URL 处理（可参考 `lib/utils/imageUrl.ts`）。
3. `评论` Sheet 为空，若需评论分析需重新从灰豚导出。

## 3. 现有千瓜入库链路（对照）

### 3.1 流程概览

```
千瓜品牌商业笔记页翻页
    ↓ 拦截 XHR（GetNoteBloggerNotes / GetBrandBusinessNoteList）
    ↓ 缓存到 noteHotMap
    ↓ 点击「AZ_推送」→ 逐条 AZ_seed
    ↓ runUploadFlow：
        1. upsert qiangua_blogger
        2. upsert qiangua_brand
        3. insert qiangua_note_info（Supabase REST，见 油猴脚本/千瓜.js）
    ↓ 另一步：拦截 GetXhsNoteUrl → 调蒲公英 note/detail
    ↓ UPDATE 笔记：XhsContent / XhsImages / XhsNoteId 等
```

相关代码与文档：

- 油猴脚本：`油猴脚本/千瓜.js`（`transformDataToDbFormat`、`runUploadFlow`、`buildNoteInfoUpdatePayload`）
- 字段映射：`.项目文档/03_技术实现/05_API接口契约/数据字段映射表.md`
- 表结构：`.项目文档/03_技术实现/03_数据库设计/创建表SQL.sql`

### 3.2 三张核心表

| 表 | 作用 |
|----|------|
| `qiangua_brand` | 品牌主数据；`BrandId` + `BrandIdKey` + `BrandName`；另有 `DateCoverage` 记录已采集日期段 |
| `qiangua_blogger` | 博主主数据，按 `BloggerId` upsert |
| `qiangua_note_info` | 笔记主表：品牌关联、投放字段、博主快照冗余、以及 `Xhs*` 详情字段 |

### 3.3 按品牌入库机制

- 千瓜 API 每条笔记带 `CooperateBindList[{ BrandId, BrandIdKey, BrandName }]`
- 入库时取第一个品牌写入笔记的 `BrandId` / `BrandName`，并 upsert `qiangua_brand`
- 下游报告、筛选、商品关联使用 **`BrandId#KF#BrandName`** 作为品牌键

### 3.4 千瓜两步才凑齐一条完整笔记

| 步骤 | 数据来源 | 得到什么 |
|------|----------|----------|
| 第 1 步：列表入库 | 千瓜 `getNoteList` | 标题、封面、互动、达人、报价、品牌、是否商业笔记等 |
| 第 2 步：详情补全 | 蒲公英 `note/detail` | `XhsNoteId`、`XhsContent`、`XhsImages`、`XhsVideo`、`XhsNoteLink` 等 |

## 4. 灰豚 → 现有库字段映射

### 4.1 CSV → 千瓜「列表 + 投放」侧

| 灰豚 CSV | 目标 DB 字段（近似） |
|----------|----------------------|
| 商业合作品牌 | `BrandName` / `CooperateBindsName` |
| 笔记提及品牌 | 可构造 `CooperateBindList` |
| 预估阅读量 | `ViewCount`（**已确认：使用 CSV 预估值，不用 XLS 阅读数**） |
| 点赞数 / 收藏数 / 评论数 / 分享数 | `LikedCount` / `CollectedCount` / `CommentsCount` / `ShareCount` |
| 预估投放报价 / 图文报价 / 视频报价 | `AdPrice` 等（**单位：元**） |
| 达人等级 | `BloggerProp` 或 `LevelName` |
| 是否参与付费推广 | `IsAdNote` |
| 商业笔记（是） | `IsBusiness` |
| 达人名称 / 粉丝数 / 小红书号 / 联系方式 | 博主冗余字段 + `LinkInfo` |
| 链接地址 | `XhsNoteLink`（库内 **`XhsNoteUrl` 几乎不用**，见 §7.5） |
| 笔记封面 | `CoverImage` |
| 发布时间 | `PublishTime` / `PubDate` / `DateCode` |
| 笔记类型（图文笔记/视频笔记） | `NoteType`（`normal` / `video`） |

### 4.2 XLS「笔记」Sheet → 千瓜「详情补全」侧

| 灰豚 XLS | 目标 DB 字段（近似） |
|----------|----------------------|
| 笔记ID | `NoteId` 和/或 `XhsNoteId` |
| 标题 / 内容 | `Title` + `XhsTitle` / **`XhsContent`**（`Content` 可留空，与千瓜一致） |
| 封面图 / 图片列表 / 视频链接 | `CoverImage` / `XhsImages` / `XhsVideo` |
| 笔记链接 | `XhsNoteLink`（主用）；`XhsNoteUrl` 可不写 |
| 阅读数 / 点赞 / 收藏 / 评论 / 分享 | **全部不用 XLS**；点赞/收藏/评论/分享见 §4.1 CSV 映射 |
| 类型 normal/video | `NoteType` |
| 标签 / IP属地 | 暂无直接列，可扩展或忽略 |

### 4.3 XLS「博主」Sheet → `qiangua_blogger`

| 灰豚 XLS | 目标 DB 字段 |
|----------|--------------|
| 用户ID / 博主ID | `XhsUserId`（**唯一稳定关联键**，见 §5.1） |
| 小红书ID | `RedId`（展示用，**不参与博主匹配**） |
| 昵称 / 头像 / 粉丝 / 个人描述 | `BloggerNickName` / `SmallAvatar` / `Fans` / `LinkInfo` 等 |
| 官方认证 | `OfficialVerified` |
| IP属地 | `Location` |

### 4.4 合并后单条记录逻辑示意（已确认原则）

**仅处理「库中尚不存在」的笔记**（按 `XhsNoteId` 判定，见 §5.1）。已入库笔记整行跳过，不写入、不更新。

```
# 主键与关联（见 §5.1）
XhsNoteId         = 灰豚笔记ID（小红书 hex，去重键）
NoteId            = 新笔记：NoteId = XhsNoteId
BloggerId         = 按 XhsUserId 查 qiangua_blogger；命中则用 DB.BloggerId，否则 BloggerId = XhsUserId
XhsUserId         = 灰豚博主用户ID（XLS「博主ID」/ 博主 Sheet「用户ID」）

# 品牌（见 §5.2）
BrandId/Key/Name  = 从 DB 品牌列表按「商业合作品牌」名称匹配

# 互动与投放（全部来自 CSV，单位：元）
ViewCount         = CSV「预估阅读量」
LikedCount 等     = CSV 点赞/收藏/评论/分享
AdPrice 等        = CSV 报价（元，与库内 AdPrice 一致）

# 内容（来自 XLS，落库习惯对齐 example.json）
Title / XhsTitle  = XLS 标题
XhsContent        = XLS 正文（Content 留空）
XhsImages / XhsVideo / XhsNoteLink = XLS
NoteIdKey         = 新笔记：XhsNoteId 前 6 位（库内均为 6 位 hex）

DataSource        = '灰豚'
```

## 5. 已确认的入库原则

### 5.1 ID 对齐：已入库跳过，仅插入新笔记

`qiangua_note_info` 表上 **`XhsNoteId`、`XhsUserId` 均已存在**（见库表 DDL）。灰豚与千瓜 ID 体系不同，对齐规则如下：

#### 笔记：按 `XhsNoteId` 去重，已存在则跳过

| 情况 | 判定 | 动作 |
|------|------|------|
| DB 中 `XhsNoteId` = 灰豚笔记 ID | 已入库 | **跳过，不导入、不更新** |
| DB 中不存在该 `XhsNoteId` | 新笔记 | **插入**；`NoteId` = `XhsNoteId`（小红书 hex） |

> 实现时建议同时检查 `NoteId` = 灰豚笔记 ID（防止少数记录主键已是 xhs id 但未写 `XhsNoteId` 的边界情况），命中任一则视为已存在并跳过。

#### 博主：仅按 `XhsUserId` 关联（唯一稳定 ID）

| 情况 | 判定 | 动作 |
|------|------|------|
| `qiangua_blogger` 中已有该 `XhsUserId` | 博主已存在 | 笔记写入时使用 DB 的 `BloggerId`（千瓜数字 ID 或历史主键） |
| 不存在该 `XhsUserId` | 新博主 | **先插入博主**；`BloggerId` = `XhsUserId`，`XhsUserId` 同步写入 |

**不用 `RedId`（小红书号）做匹配**——仅 `XhsUserId`（灰豚 XLS「博主ID」/ 博主 Sheet「用户ID」，24 位 hex）稳定可靠。

笔记表冗余字段 `qiangua_note_info.XhsUserId` 与博主表保持一致，便于报告、矩阵等下游查询。

#### 品牌

| 字段 | 规则 |
|------|------|
| `BrandId` / `BrandIdKey` / `BrandName` | 从 DB 拉取 `qiangua_brand`，按 CSV「商业合作品牌」名称精确匹配 |

**合并时先读库：**

1. `qiangua_brand` 全量
2. `qiangua_note_info`：`NoteId`, `XhsNoteId`（构建已入库笔记集合）
3. `qiangua_blogger`：`BloggerId`, `XhsUserId`（构建 `XhsUserId` → `BloggerId` 映射）

本地 join 灰豚 CSV + XLS 后，标记每行 `import_action`：`skip`（已存在）或 `insert`（新笔记）。

### 5.2 品牌匹配与剔除规则（已确认）

**匹配方式：** CSV 列「商业合作品牌」与 `qiangua_brand.BrandName` **精确匹配**（不用文件名映射）。

**主动剔除：** 商业合作品牌名称含 **「美妆」「香氛」「香水美妆」** 的子线 **不入库**（`skip_reason = 剔除子线（美妆/香氛）`）。

| 灰豚「商业合作品牌」 | 条数 | 处理 |
|---------------------|------|------|
| `Prada普拉达` | 70 | ✅ 入库 |
| `Prada普拉达香水美妆` | 190 | ❌ 剔除 |
| `LOEWE罗意威` | 83 | ✅ 入库 |
| `LOEWE罗意威香氛` | 19 | ❌ 剔除 |
| `FERRAGAMO菲拉格慕` | 38 | ✅ 入库（需 DB 有同名 BrandName） |
| `Ferragamo菲拉格慕美妆` | 24 | ❌ 剔除 |
| `LoroPiana 诺悠翩雅` | 31 | ✅ 入库 |
| `Bottega Veneta` | 3 | ✅ 入库 |

文件名（如 `Prada _商业笔记_...csv`）仅表示灰豚导出时的筛选范围，**入库归属以 CSV「商业合作品牌」为准**；同一文件内主品牌与子线并存时，靠上述剔除规则分开。

「笔记提及品牌」列可写入 `CooperateBindList` 辅助信息；笔记表 `BrandId` / `BrandName` 来自匹配后的主品牌。

### 5.3 互动数口径（已确认：全部 CSV）

| 目标字段 | 灰豚来源 | 说明 |
|----------|----------|------|
| `ViewCount` | CSV `预估阅读量` | |
| `LikedCount` | CSV `点赞数` | |
| `CollectedCount` | CSV `收藏数` | |
| `CommentsCount` | CSV `评论数` | |
| `ShareCount` | CSV `分享数` | |
| `LikeCollect` | 可推导 | `点赞数 + 收藏数`（可选） |

XLS 中的阅读数、点赞等**仅作本地核对**，不入库。

### 5.4 报价单位（已确认：元）

`AdPrice` 及 CSV 中的 `预估投放报价` / `图文报价` / `视频报价` 单位均为 **元**，与库内 `qiangua_note_info.AdPrice`（integer）一致，**无需换算**。

### 5.5 已入库数据：只跳过、不更新（已确认）

- 笔记：`XhsNoteId` 已存在于 `qiangua_note_info` → **整行跳过**
- 不覆盖千瓜已写入的 `Content`、`XhsContent`、`Ai*` 等字段
- 灰豚的价值是**补全千瓜未采集到的新笔记**，而非刷新历史数据

### 5.6 数据来源标记 `DataSource`（待执行 migration）

在以下表中新增 `DataSource` 字段（VARCHAR），灰豚导入记录填 **`灰豚`**；千瓜历史数据可为 `千瓜` 或 NULL：

| 表 | 说明 |
|----|------|
| `qiangua_note_info` | 笔记列表，每条灰豚导入笔记标记 `灰豚` |
| `qiangua_blogger` | 博主列表，每条灰豚新建博主标记 `灰豚` |

便于后续筛选、统计、排查数据来源。`qiangua_brand` 暂不增加（品牌主数据仍为千瓜体系，仅做名称映射）。

**DDL 脚本：** [`scripts/migration-add-data-source.sql`](../scripts/migration-add-data-source.sql)

```sql
-- 列名遵循库内 PascalCase 惯例，非 snake_case data_source
ALTER TABLE qiangua_note_info ADD COLUMN IF NOT EXISTS "DataSource" VARCHAR(32);
ALTER TABLE qiangua_blogger  ADD COLUMN IF NOT EXISTS "DataSource" VARCHAR(32);
-- 允许值：'千瓜' | '灰豚' | NULL（历史数据）
```

**实施顺序：** 阶段 B 批量入库前在 Supabase SQL Editor 执行上述脚本；汇总表预览阶段即带上 `DataSource` 列。

### 5.7 灰豚缺失、千瓜有的字段

可置空或由规则推导：

- `NoteIdKey`、`SpreadScore`、`Index`、`Lcc`、`LikeCollect`
- `McnName`、`LevelNumber`（薯等级，如金冠薯）
- `DateCoverage`（品牌表上的已采集日期段，导入后可选手动更新）

`DateCode` 可由 `PublishTime` 推导为 `YYYYMMDD`。

### 5.8 灰豚已有、千瓜需第二步采集的字段

正文、图片列表、视频、标签、IP 属地——**首次入库即可写入 `Content` / `XhsContent` / `XhsImages` 等**，无需再走蒲公英详情流程。

## 6. 千瓜 vs 灰豚对比

| 维度 | 千瓜插件 | 灰豚离线 |
|------|----------|----------|
| 采集方式 | 网页翻页 + API 拦截 | 平台导出文件 |
| 速度 | 慢（逐页、逐条详情） | 快（批量） |
| 写入目标 | `qiangua_*` 三张表 | 相同三张表 |
| 详情补全 | 需第二步蒲公英 API | XLS 已含 |
| 品牌 ID | 千瓜原生 `BrandId` | 需名称 → `BrandId` 映射 |
| 笔记主键 | 千瓜数字 `NoteId` | 小红书笔记 ID |
| 评论数据 | 未统一入库 | 当前导出为空 |

**结论：** 不是换数据库，而是**换数据源 + 增加 transform 层**；灰豚 CSV + XLS 在语义上已覆盖千瓜「列表 + 详情」两步的并集。

## 7. 入库流程：先本地汇总，确认后批量写入

### 7.1 两阶段流程（已确认）

```
阶段 A：本地汇总（staging，不入库）
  1. 从 DB 拉取：品牌列表、已有笔记（XhsNoteId→NoteId）、已有博主（XhsUserId→BloggerId）
  2. 读取 5 个品牌 CSV + 小红书笔记详情.xls，按笔记 ID join
  3. 按 §5 规则补全 NoteId / BloggerId / BrandId / DataSource 等
  4. 输出一张汇总表（建议 xlsx/csv，放本目录，如 `staging_灰豚汇总_待确认.xlsx`）
  5. 人工检查：品牌映射未命中、主键命中/新建数量、异常字段

阶段 B：确认后批量入库（仅 insert，不 update 已有笔记）
  6. migration：qiangua_note_info、qiangua_blogger 增加 DataSource 列（若尚未添加）
  7. 仅导入汇总表中 import_action = insert 的行
  8. 写入顺序：新博主（insert）→ 新笔记（insert）；brand 不新建
  9. 可选：更新 qiangua_brand.DateCoverage
```

### 7.3 已确认事项（2026-06-30）

| # | 事项 | 结论 |
|---|------|------|
| 1 | 互动数口径 | **全部用 CSV**（阅读/点赞/收藏/评论/分享） |
| 2 | `AdPrice` 单位 | **元**，与库内一致，直接写入 |
| 3 | 已入库笔记 | **`XhsNoteId` 命中则跳过**，不更新 |
| 4 | 博主匹配 | **仅用 `XhsUserId`**；`RedId` 不参与匹配；新博主 `BloggerId` = `XhsUserId` |

### 7.4 实现注意点

1. **`qiangua_note_info` 去重集合**：除 `XhsNoteId` 外，建议把 `NoteId` 也纳入（部分历史行可能主键已是 xhs id 但 `XhsNoteId` 为空）。**库内样本（`example.json` 100 条）显示 `XhsNoteId` 100% 已填**，以 `XhsNoteId` 去重即可覆盖千瓜历史笔记。
2. **新博主 `BloggerIdKey`**：`qiangua_blogger` 该字段 NOT NULL；库内样本 **100% 有 6 位 `BloggerIdKey`**（如 `97ad2c`）。新博主可用 `XhsUserId` 前 6 位生成；`NoteIdKey` 同理（nullable 但建议同样生成）。
3. **千瓜博主无 `XhsUserId` 的边界**：若达人仅以千瓜数字 `BloggerId` 存在、尚未写入 `XhsUserId`，灰豚会按新博主插入。样本中笔记表 **`XhsUserId` 100% 已填**，优先从 `qiangua_blogger` / `qiangua_note_info` 构建 `XhsUserId → BloggerId` 映射即可。
4. **默认值**：新笔记 `AiStatus` = `待分析`；`XhsNoteInvalid` = false。
5. **汇总表首要指标**：458 条中 **skip 多少 / insert 多少**——这决定灰豚本次实际增量。

### 7.5 库内样本校验（`example.json`，100 条）

自 `qiangua_note_info` 导出的 100 条样本，用于核对字段习惯与方案可行性。

#### ID 与去重（✅ 方案成立）

| 字段 | 样本表现 | 对方案的影响 |
|------|----------|--------------|
| `NoteId` | **100% 千瓜数字 ID**（如 `1701578494`） | 历史笔记主键均为数字；**skip 时保留原 `NoteId`，不改为 xhs id** |
| `XhsNoteId` | **100% 已填**，24 位 hex；与 `NoteId` **从不相同** | **去重键用 `XhsNoteId` 正确**；灰豚笔记 ID 与库内 `XhsNoteId` 比对即可 |
| `BloggerId` | **100% 千瓜数字 ID** | 命中博主时用库内数字 `BloggerId`，不用 xhs id 覆盖 |
| `XhsUserId` | **100% 已填**（笔记冗余字段） | 博主关联键可靠；可从样本构建 `XhsUserId → BloggerId`（100 条中 94 对唯一映射） |

#### 正文与链接（⚠️ 需调整映射）

| 字段 | 样本填充率 | 灰豚应写入 |
|------|------------|------------|
| `Content` | **0/100** | 可不写或留空（与千瓜习惯一致） |
| `XhsContent` | **100/100** | XLS 正文写这里 |
| `Title` / `XhsTitle` | 99/100 | 两者同步写 XLS 标题 |
| `XhsNoteLink` | **100/100** | CSV/XLS 笔记链接写这里 |
| `XhsNoteUrl` | **0/100** | 非主用字段，可不写 |
| `RedId` | **0/100** | 可从 CSV「小红书号」补全（增量价值） |

#### 品牌与 CooperateBindList（✅ 方案成立）

- 样本品牌名：`Balenciaga`、`GUCCI`、`Prada普拉达`（与灰豚子线 `Prada普拉达` **一致**）
- `CooperateBindList` 示例：`[{ BrandId: 31731, BrandName: "Prada普拉达", BrandIdKey: "6eb6a5" }]`（导出为 JSON 字符串，入库需 **JSONB 数组**）
- 灰豚另含 `Prada普拉达香水美妆` 等子线，需与 `qiangua_brand` 全量列表逐条匹配（不能仅靠这 100 条样本）

#### 数值口径（✅ AdPrice；⚠️ ViewCount 语义）

| 字段 | 样本（Prada 子集） | 灰豚 CSV | 结论 |
|------|-------------------|----------|------|
| `AdPrice` | 4000–16000 元量级 | 预估报价 median 5600 | **单位为元，直接写入** ✅ |
| `ViewCount` | 621–4595（Prada）；全样本 max 629361 | 预估阅读 median 9807 | 已确认写 CSV；注意千瓜多为**实际阅读/监测口径**，灰豚为**预估阅读**，同字段混存时下游对比需知情 |

#### 笔记类型与其它

- `NoteType`：`normal` 70 / `video` 30 — 与灰豚 XLS 一致；CSV「图文笔记」→ `normal`，「视频笔记」→ `video`
- `IsAdNote`：69 true / 31 false — 对应 CSV「是否参与付费推广」
- `LikeCollect` / `Lcc`：样本多为空 — 灰豚可不填或由 CSV 点赞+收藏推导
- **无 `DataSource` 列** — migration 仍需要

#### 与当前灰豚 458 条的重叠（样本仅 100 条）

- 笔记 `XhsNoteId` 重叠：**0**（样本为 2025 年 Balenciaga/GUCCI/Prada，灰豚为 2026-04~06 另一批）
- 博主 `XhsUserId` 重叠：**12**（说明部分达人跨批次复用，命中时应沿用数字 `BloggerId`）

**结论：** 当前方案整体可行；需微调字段落点（`XhsContent`/`XhsNoteLink`、新记录 `NoteIdKey`/`BloggerIdKey`），去重与博主关联策略与库内真实数据一致。
